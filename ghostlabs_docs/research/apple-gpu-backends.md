> Investigation report supporting `ghostlabs_docs/plan.md` (2026-07-02).

# GPU compute on Apple M-series from Python (as of July 2026) — porting a CuPy + CUDA SSSP autorouter

## 0. What the port actually needs (from this repo)

`orthoroute/algorithms/manhattan/pathfinder/cuda_dijkstra.py` (and `persistent_kernel.py`, `via_kernels.py`, etc.) use:
- `atomicMin` on `unsigned long long` keys packing float-cost bits + parent index (`pack_key(g,p) = (f2u(g)<<32) | p`, line 218-219; native `atomicMin(&dist_parent[...], new_key)` line 191; CAS-emulated `atomicMin64` lines 222-231 for the packed relax at line 371)
- CAS-based `atomicMinFloat` (line 201, 446, 668)
- bit-packed frontier words, wavefront expansion, per-ROI batched grids (`(K, max_roi_size)` layouts)
- CuPy array math on int32/float32 arrays (tens of millions of elements), plus a persistent-kernel variant (needs device-wide sync — no Metal equivalent, see §7).

## 1. MLX (Apple) — **recommended primary**

- **Current version:** mlx 0.31.2 on PyPI, released 2026-04-22; Python 3.10-3.14; macOS arm64 wheels require macOS ≥ 14; also Linux wheels + CUDA extras (`mlx[cuda]`, `cuda12`, `cuda13`). https://pypi.org/project/mlx/ . Actively maintained (releases every ~3-4 weeks through 2026): https://github.com/ml-explore/mlx/releases
- **NumPy-likeness:** `mlx.core` is deliberately NumPy-like (docs: https://ml-explore.github.io/mlx/). Dtypes: int8/16/32/64, uint8/16/32/64, float16/bfloat16/float32, bool, complex64; **float64 is CPU-only (GPU raises)** — https://ml-explore.github.io/mlx/build/html/python/data_types.html . Key gaps vs NumPy (all from https://ml-explore.github.io/mlx/build/html/usage/indexing.html): **no boolean-mask reads, no `nonzero()`, no single-arg `where()`** (no data-dependent output shapes); boolean-mask *assignment* works; integer/array indexing, `take`/`take_along_axis` work; GPU indexing is unchecked (OOB = UB). Frontier compaction must be done cumsum/scatter-style or in a custom kernel — which the CUDA code already does with bit-packed frontier words, so this is not a blocker.
- **Lazy evaluation:** ops build a graph; `mx.eval()` (or `.item()`, NumPy conversion, printing) materializes. Batch per outer loop iteration. https://ml-explore.github.io/mlx/build/html/usage/lazy_evaluation.html
- **Unified memory / NumPy interop:** buffer protocol + DLPack; `np.array(a, copy=False)` gives a zero-copy view. https://ml-explore.github.io/mlx/build/html/usage/numpy.html
- **`mx.fast.metal_kernel`** (https://ml-explore.github.io/mlx/build/html/python/_autosummary/mlx.core.fast.metal_kernel.html, guide: https://ml-explore.github.io/mlx/build/html/dev/custom_metal_kernels.html): you write the **body** of an MSL kernel as a string (plus arbitrary `header` MSL — so any MSL intrinsic is reachable); auto-generated signature from `input_names`/`output_names`, template params (dtypes/ints/bools), auto-passed `*_shape/*_strides/*_ndim`, `grid`/`threadgroup` given per call, `ensure_row_contiguous`, JIT-compiled and cached per kernel object. `atomic_outputs=True` types outputs as `device atomic<T>` and `init_value` pre-fills outputs; the official docs' advanced example uses `atomic_fetch_add_explicit` + simdgroup pre-reduction. Limitations: dispatch grid is set from the host per call (no indirect/device-side dispatch), outputs are freshly allocated (functional style → ping-pong buffers), and 64-bit `atomic_ulong` outputs are not an officially documented path (the `header`+raw-body escape hatch makes it expressible; must be validated with a 20-line prototype).
- **Maturity for non-ML:** good and improving — mlx-graphs GNN library (https://github.com/mlx-graphs/mlx-graphs), community custom-kernel projects (e.g. https://github.com/manishklach/mlx-metal-kernels, TurboQuant KV-cache kernels write-up 2026). It is still an ML-first framework: no sparse arrays, no cooperative launch, kernels go through the lazy graph.
- **Strategic bonus:** MLX gained a **CUDA backend** (Linux/NVIDIA, `pip install "mlx[cuda]"`, in preview since 2025 — https://github.com/ml-explore/mlx/discussions/2422): one MLX code path could eventually serve both Mac users and the existing NVIDIA user base (custom Metal kernels still need CUDA twins, but the array-math layer becomes shared).

## 2. PyTorch MPS (evaluated for completeness; upstream maintainer bans it as a dependency)

- MPS backend is mature and actively invested (docs: https://docs.pytorch.org/docs/stable/notes/mps.html, PyTorch 2.12; RFC to rename MPS→Metal backend: https://github.com/pytorch/pytorch/issues/173225). int32/int64 etc. supported.
- **`torch.mps.compile_shader(source)`** compiles MSL from Python and dispatches kernels on MPS tensors — the closest thing to CuPy `RawKernel` in PyTorch (stable docs: https://docs.pytorch.org/docs/stable/generated/torch.mps.compile_shader.html; documented since ~2.10, 2025).
- `scatter_reduce(amin)` on MPS has a history of gaps/bugs (https://github.com/pytorch/pytorch/issues/97093); `PYTORCH_ENABLE_MPS_FALLBACK=1` CPU fallback exists but is a performance cliff (https://github.com/pytorch/pytorch/issues/134416). Technically viable overall, but excluded by project policy.

## 3. wgpu-py — **recommended fallback / portability play**

- v0.31.1 (June 2026), maintained by pygfx; wraps wgpu-native (Rust wgpu); Metal on macOS; WebGPU coverage "complete enough to build pygfx", ~100% test coverage. https://github.com/pygfx/wgpu-py , docs https://wgpu-py.readthedocs.io/
- **Standard WGSL atomics are 32-bit only** (`atomic<i32|u32>`): https://www.w3.org/TR/WGSL/ . The WebGPU-standard 64-bit min/max proposal is still open (opened 2025-02-14, "Waiting for PR"): https://github.com/gpuweb/gpuweb/issues/5071
- **But wgpu native extensions rescue the 64-bit trick:** wgpu-native exposes `WGPUNativeFeature_ShaderInt64` (0x00030026), `ShaderInt64AtomicMinMax` (0x0003002C), `ShaderInt64AtomicAllOps` (0x0003002D) (https://github.com/gfx-rs/wgpu-native/blob/trunk/ffi/wgpu.h), and wgpu's docs state `SHADER_INT64_ATOMIC_MIN_MAX` is available **on Metal on Apple9, or Apple8 + Mac2, requiring MSL 2.4+** (https://docs.rs/wgpu/latest/wgpu/struct.Features.html). wgpu-py lets you request native features/limits by kebab-case name via `request_device_sync(required_features=[...], required_limits={...})` (https://wgpu-py.readthedocs.io/en/stable/backends.html). Needs a prototype to confirm the naga WGSL `atomic<u64>` path end-to-end from Python.
- **Buffer limits:** WebGPU defaults are small (maxStorageBufferBindingSize default 128 MiB — https://www.w3.org/TR/webgpu/#limits); you must request higher limits up to adapter limits for tens-of-millions-of-element arrays (a 50M-int32 array is 200 MB).
- **Cost:** no NumPy-like array layer at all — every reduction/compaction/elementwise op becomes a hand-written WGSL kernel; explicit buffer/bind-group management. Precedent: SlicerWGPU (https://github.com/pieper/SlicerWGPU).

## 4. Taichi — not viable

Development halted ~Aug 2024; community discussion "Development on Taichi has halted" (https://github.com/taichi-dev/taichi/discussions/8506); last releases v1.7.x in 2024 with only a maintenance PyPI push mid-2025 (https://github.com/taichi-dev/taichi/releases). Metal backend exists but is second-tier (sparse SNode support removed; GGUI never supported Metal — https://github.com/taichi-dev/taichi/issues/7477). Do not build on it.

## 5. jax-metal — dead

Last release 0.1.1 on 2024-10-08 (https://pypi.org/project/jax-metal/); experimental, no float64; broken with recent JAX/macOS (https://github.com/jax-ml/jax/issues/34109); JAX maintainers closed jax-metal issues in Dec 2025 citing no active development; Apple unresponsive (https://developer.apple.com/forums/thread/743350). A community "jax-mps" is embryonic (https://github.com/jax-ml/jax/discussions/34648). Exclude.

## 6. Everything else

- **Mojo/MAX (Modular):** Apple Silicon GPU support announced 2025-09-21 (nightlies): Mojo→AIR bitcode→metallib; M1-M5; macOS 15+, Xcode 16+; **atomics and barriers were explicitly not yet functional at launch**, on the roadmap (https://forum.modular.com/t/apple-silicon-gpu-support-in-mojo/2295). Modular Platform 26.1 (2026) graduates the MAX Python API and expands Apple GPU support (https://docs.modular.com/max/faq/, https://github.com/modular/modular). Watch, don't bet on it yet for atomic-heavy SSSP.
- **cuPyNumeric (ex-cuNumeric):** macOS arm64 wheels are **CPU-only**; GPU support = Linux/NVIDIA (https://docs.nvidia.com/cupynumeric/latest/installation.html). Not an option.
- **Vulkan-on-Mac (kompute/MoltenVK):** MoltenVK does **not** support `VK_KHR_shader_atomic_int64` (open issue https://github.com/KhronosGroup/MoltenVK/issues/1692) — the packed-64-bit trick dies there. Exclude.
- **Direct Metal from Python:** `metalcompute` (py-metal-compute) 0.2.x — tiny single-author lib to run MSL kernels on macOS ≥14 (https://pypi.org/project/metalcompute/, https://github.com/baldand/py-metal-compute); PyObjC gives full Metal API. Useful as an escape hatch for things MLX's kernel API can't express (indirect dispatch, MTLSharedEvent loops), not as the main framework. **tinygrad** has a production Metal backend driven from Python (https://github.com/tinygrad/tinygrad/blob/master/tinygrad/runtime/ops_metal.py) — precedent, but its kernel model is its own IR, not raw-MSL-friendly.

## 7. Metal Shading Language / hardware facts that shape the kernel port

All from the **Metal Feature Set Tables, May 21, 2026 edition** (https://developer.apple.com/metal/Metal-Feature-Set-Tables.pdf) unless noted:
- Family map (p.2): **M1 = Apple7, M2 = Apple8, M3/M4 = Apple9, M5 = Apple10**.
- **"64-bit atomics" row: Apple9** (p.5); footnote 7 (p.6): *"GPU devices in the Apple8 family support 64-bit atomic minimum and maximum using ulong, on both buffers and textures, only on macOS. The full set of 64-bit atomic operations is supported on all platforms starting with Apple9."* → the exact primitive this codebase needs (`atomicMin` on packed u64) is **native on M2/M3/M4/M5, absent on M1** (M1 also lacks 64-bit CAS, so the CUDA `atomicMin64` CAS-emulation cannot be ported to M1 either).
- 32-bit atomics (`atomic_int/uint` fetch_min/max/add/CAS, relaxed order) are baseline everywhere; **"Floating-point atomics": Apple7+** (p.5); "64-bit integer math" (non-atomic long/ulong ALU): Apple3+ (p.4). Metal 3.1 added ulong min/max **texture** atomics (see MoltenVK discussion https://github.com/KhronosGroup/MoltenVK/discussions/1938); current MSL spec is 4.1 (https://developer.apple.com/metal/Metal-Shading-Language-Specification.pdf).
- Threadgroups: max 1024 threads, **32 KB threadgroup memory** (p.7); SIMD-scoped permute (Apple6+) and reduction (Apple7+) ops exist (p.4) — good for pre-reducing before atomics.
- **No cooperative-groups / device-wide sync**: nothing like `grid.sync()`; the `persistent_kernel.py` design cannot port. Standard practice: one dispatch per relaxation wave, multiple dispatches encoded in one command buffer (Metal guarantees sequential dependency within a queue), `MTLDispatchType` / events for cross-buffer sync. Indirect dispatch (`dispatchThreadgroups(indirectBuffer:)`) exists in Metal but is *not* exposed by `mx.fast.metal_kernel` — use capped fixed grids with early-exit, or drop to metalcompute/PyObjC for that one loop.
- Max buffer size is a runtime property `MTLDevice.maxBufferLength` (referenced in footnote 9, p.10) — on M-series Macs it is multi-GB, far beyond the ~0.2-1 GB arrays here; unified memory means no H2D/D2H copies at all.

## 8. Real-world precedents

- **mlx-graphs** — graph (GNN) workloads on MLX/Apple Silicon, up to ~10x vs other frameworks on large datasets (https://github.com/mlx-graphs/mlx-graphs).
- **tinygrad** Metal backend — full DL framework driving Metal from Python for years.
- **SlicerWGPU** — scientific compute via wgpu-py (https://github.com/pieper/SlicerWGPU).
- Community MLX custom-kernel projects (flash-attention-style kernels, KV-cache quantization) show `mx.fast.metal_kernel` handling nontrivial hand-written kernels.
- **No known GPU EDA router / batched-SSSP project on Metal from Python** — academic GPU shortest-path work is CUDA/OpenGL (e.g. UC Merced SPM shaders, http://graphics.ucmerced.edu/papers/14-mig-spm.pdf). This port would be first-of-kind; expect to write, not find, the SSSP kernels.

## 9. Recommendation

**(a) Array-math layer → MLX (`mlx.core`), primary.** Closest CuPy analogue that is actively maintained on Apple GPUs; unified memory + zero-copy NumPy views; int32/float32 everywhere (never float64 on GPU). Design around: lazy eval (`mx.eval` once per routing iteration), no boolean-mask reads/`nonzero` (replace with `where` + cumsum/scatter compaction or in-kernel bit-packed frontiers, which the CUDA code already uses). Fallback for stray host-side logic: NumPy on CPU is nearly free thanks to unified memory.

**(b) Custom SSSP kernels → `mx.fast.metal_kernel` (MSL), primary; wgpu-py (WGSL + native `shader-int64-atomic-min-max`) fallback.**
- M2/M3/M4/M5 (Apple8/9/10, macOS): keep the packed dist+parent design, but use **native** `atomic_fetch_min_explicit` on `atomic_ulong` — simpler than the CUDA CAS-emulation path.
- M1 (Apple7): no 64-bit atomics of any kind → ship a 32-bit variant: `atomic_fetch_min_explicit` on monotonic float-bit `uint` distance + a second "parent resolution" pass (any relaxing thread whose candidate equals the winning distance writes the parent; benign race), or gate M1 out and require M2+.
- Runtime gating: probe `supportsFamily(.apple8)`-equivalent at startup (via a tiny compile-test kernel or device-info query) and select kernel variant.
- Replace the persistent kernel with wave-per-dispatch batched inside one command buffer; cap grid sizes to avoid per-wave host readbacks.
- Keep kernels in raw MSL strings structured so the same algorithm has a CUDA-C twin — MLX's CUDA backend plus CuPy parity keeps the NVIDIA path first-class for upstream.
- If MLX's kernel API proves too confining (indirect dispatch, event-driven loops), drop just that loop to `metalcompute`/PyObjC rather than switching frameworks.
- Avoid: PyTorch (banned upstream), Taichi (halted), jax-metal (dead), Vulkan/MoltenVK (no 64-bit atomics), cuPyNumeric (no Apple GPU); watch Mojo/MAX (atomics not ready at 2025 launch, improving through 26.1).

## Recommendations

- Primary backend: MLX 0.31.2+ — mlx.core for the NumPy-like int32/float32 array layer and mx.fast.metal_kernel for the hand-written SSSP kernels; it is the only Apple-GPU option that is simultaneously actively maintained, CuPy-shaped, unified-memory zero-copy with NumPy, and able to run raw MSL with atomics.
- Kernel port, M2 and newer (Apple8/9/10 on macOS): keep the packed dist+parent u64 design and use native atomic_fetch_min_explicit on atomic_ulong (declared via the metal_kernel header/body escape hatch); this is simpler than the CUDA CAS-emulation. Gate at startup with a compile-probe kernel and require macOS.
- Kernel port, M1 (Apple7): no 64-bit atomics of any kind (not even CAS) — ship a 32-bit fallback: atomic_fetch_min_explicit on monotonically-mapped float-bit uint distances plus a second parent-resolution pass (benign race: any relaxer matching the winning distance writes the parent), or officially require M2+.
- Replace the persistent-kernel/device-wide-sync design: Metal has no cooperative-groups equivalent; encode one dispatch per relaxation wave, batching many waves into a single command buffer, with capped fixed grids and early-exit threads to avoid per-wave host readbacks (mx.fast.metal_kernel cannot do indirect dispatch; if that one loop needs indirect dispatch or MTLSharedEvent, drop just that loop to metalcompute/PyObjC).
- Array-layer porting rules for MLX: batch work and call mx.eval() once per routing iteration (lazy evaluation); never use float64 on GPU; replace boolean-mask reads / nonzero() with mx.where + cumsum/scatter compaction or in-kernel bit-packed frontiers (the CUDA code already uses bit-packed frontier words, so this maps naturally).
- Fallback backend: wgpu-py 0.31.1+ with WGSL kernels, requesting native features 'shader-int64' and 'shader-int64-atomic-min-max' (supported on Metal for Apple8+Mac2 / Apple9 per wgpu docs) plus raised max-storage-buffer-binding-size/max-buffer-size limits; prototype the naga atomic<u64> path early since it is a native extension, and budget for hand-writing every reduction/compaction kernel (no array layer).
- Build a 1-day spike before committing: (1) mx.fast.metal_kernel with atomic_ulong min on an M2/M3, (2) the same relax kernel in wgpu-py with shader-int64-atomic-min-max, (3) measure a 10M-node wavefront iteration; the results decide primary vs fallback definitively.
- Keep the CUDA/CuPy path first-class: structure kernels as paired MSL/CUDA-C strings behind one dispatch interface, and consider migrating the array layer itself to MLX everywhere long-term since MLX now has a CUDA backend (mlx[cuda]) — one array API for both platforms.
- Do not invest in: PyTorch MPS (technically workable via torch.mps.compile_shader but banned by the upstream maintainer), Taichi (development halted 2024), jax-metal (unmaintained since Oct 2024, broken on current macOS/JAX), Vulkan/MoltenVK/kompute (no VK_KHR_shader_atomic_int64), cuPyNumeric (CPU-only on Mac). Re-evaluate Mojo/MAX in 12 months once GPU atomics/barriers and the Python custom-op path mature (26.x line).
